# Claude Codeに「Flutterビルドして」と頼んだら毎回違うことをされた

Claude Codeに「FlutterプロジェクトをiOSシミュレーターでビルドして、UIを確認できる状態にして」と指示しました。シンプルな依頼のはずが、毎回異なるアプローチを取られ、まともに動く状態になりませんでした。

具体的に起きた問題：

- **ブラウザ（Chrome）向けにビルドしてしまった**（iOS向けと言ったのに）
- **環境変数を勝手に追加した**（成功した前提で設定を書き加えた）
- **ローカルで実行せず手順だけ示した**（「あなたが実行してください」と言い直す羽目に）
- **ビルド実行時にアプリがクラッシュ**（エラーを無視して次に進んだ）
- **同じコマンドなのに再実行で失敗**（シミュレーターは起動するがアプリは黒い画面）

毎回同じプロンプトで指示しても、Claudeが異なるアプローチを取ったり、途中のステップを飛ばしたり、エラーを見逃して次に進んでしまう。**再現性がゼロ**でした。

## なぜプロンプト指示では駄目だったのか

根本的な原因は3つあります。

**1. LLMは毎回異なる判断をする**
同じ「Flutterをビルドして」という指示でも、ある時はWeb向け、ある時はiOS向け、ある時は手順の提示だけ、とアプローチがブレます。

**2. エラーを自己判断で「問題なし」として進む**
ビルドエラーが出ても「軽微な警告」と判断して次のステップに進むことがありました。人間なら確認するところを、AIが勝手にスキップしてしまう。

**3. バックグラウンド実行のエラー検知を怠る**
`flutter run`をバックグラウンドで実行した場合、出力確認の手段（`TaskOutput`）はあるのですが、実際にはそれを使わずエラーを見逃していました。

つまり、**複数ステップにまたがる作業をプロンプトで指示すると、どこかで判断ミスが入り込む**という問題です。

## Claude Code Skillsとは

この問題を解決したのが、Claude Codeの**Skills**機能です。

Skillsは、`.claude/skills/`ディレクトリ配下に`SKILL.md`ファイルを配置すると、`/スキル名`というスラッシュコマンドでClaude Codeから呼び出せる仕組みです。

```
.claude/skills/mock-build/
├── SKILL.md              # スキルの定義
└── scripts/
    └── run-mock-build.sh  # 実行されるスクリプト
```

SKILL.mdにはフロントマターでスキルの設定を記述します。

```yaml
---
name: mock-build
description: iOSシミュレーターでFlutterアプリをモックデータモードでビルド・実行する。
allowed-tools: Read, Bash
user-invocable: true
---
```

| フィールド | 説明 |
|---|---|
| `name` | スキル名。`/mock-build`で呼び出せる |
| `description` | Claudeがいつこのスキルを使うべきか判断するための説明 |
| `allowed-tools` | このスキルが使えるツール |
| `user-invocable` | `true`でユーザーがスラッシュコマンドから直接呼び出し可能 |

SKILL.mdの本文には、スクリプトの実行指示とClaude自身の行動ルールを書きます。ここがSkillsの強力なポイントで、後述します。

## Skills設計で考えたこと

今回のSkills設計では、4つの重要な判断をしました。いずれも**実際の失敗体験**に基づいています。

### 判断1: 全ステップ同期実行にする

`flutter run`はホットリロードのために終了しないプロセスです。Claude CodeのBashツールにはデフォルト2分のタイムアウトがあるため、`run_in_background: true`を使えばタイムアウトを回避できます。

しかし、バックグラウンド実行にすると**Claudeがエラーを検知できない**問題が実際に起きていました。出力を確認する手段はあるのに、それを使わずにタスクを進めてしまう。SKILL.mdにルールとして書いても、確実に従う保証はありません。

**結論：ホットリロードは諦めて、全ステップ同期実行にする。** コード変更時は再度`/mock-build`を実行する運用としました。

### 判断2: flutter runをbuild + install + launchに分離する

`flutter run`は内部で以下を一括実行します。

```
flutter run
  ├── pub get（依存関係の解決）
  ├── Xcodeビルド
  ├── .appファイルをシミュレーターにインストール
  ├── アプリを起動
  └── デバッグ接続を確立（ホットリロード用）
```

これを一括で実行すると、**どのステップで失敗したかわからない**。そこで以下のように分離しました。

```bash
# ビルド（同期実行、成否を確認）
flutter build ios --simulator --dart-define=USE_MOCK_DATA=true

# インストール（simctlで直接操作）
xcrun simctl install <UUID> <.appパス>

# 起動（simctlで直接操作）
xcrun simctl launch <UUID> <bundle-id>
```

各ステップが独立しているため、失敗箇所が明確になります。

### 判断3: 不要なステップを削る

当初は「Flutterデバイス認識確認（`flutter devices`）」というステップを含めていました。しかし実際に動かすと、ワイヤレスデバイス検索の遅延でgrepのタイミングが合わず失敗しました。

後続ステップの依存関係を整理した結果、**どのステップもFlutterのデバイス認識に依存していない**ことがわかりました。

- `flutter build ios --simulator` → シミュレーター用にビルドするだけ。デバイス認識不要
- `xcrun simctl install` → simctlにUUIDを直接渡す。Flutter不要
- `xcrun simctl launch` → 同上

ステップ自体が不要だったので削除しました。動かしてみて初めてわかることもあります。

### 判断4: SKILL.mdにClaudeの行動ルールを書く

SKILL.mdの本文には、スクリプトの実行指示だけでなく、**エラー時のClaudeの振る舞い**を明記しました。

- スクリプトがエラー終了した場合、出力内容をユーザーにそのまま報告すること
- 自動修正は行わず、ユーザーの指示を仰ぐこと
- ビルド成功を自分で判断しない。スクリプトの終了コードとログのみを信頼すること

これにより、Claudeがエラーを見逃して次に進む問題を防止しています。Skillsの本質は**スクリプトの実行だけでなく、AIの行動を制約できる点**にあります。

## 実装の概要

最終的なスクリプトは7ステップ構成です。

| ステップ | 処理 | 詳細 |
|---|---|---|
| 1 | 既存プロセスの停止 | flutter runの残存プロセスを検知・停止 |
| 2 | Simulator.app起動確認 | 未起動なら起動 |
| 3 | デバイスのboot確認 | 未bootならboot |
| 4 | flutter pub get | 依存関係の解決 |
| 5 | ビルド | `flutter build ios --simulator`で同期実行 |
| 6 | インストール | `xcrun simctl install`で.appをインストール |
| 7 | 起動 | `xcrun simctl launch`でアプリ起動 |

スクリプト冒頭の`set -euo pipefail`により、どのステップで失敗しても即座にスクリプトが終了します。ステップ1〜3で**現在の状態を毎回チェック**するため、シミュレーターが起動済みでも未起動でも正しく動作します。

## まとめ

### Skills化で何が変わったか

| 課題 | Skills化で解決 |
|---|---|
| 毎回異なるアプローチを取られる | スクリプトとして手順が固定される |
| エラーを見逃して次に進む | `set -euo pipefail`で即終了 + SKILL.mdで報告ルールを明記 |
| 毎回長いプロンプトが必要 | `/mock-build`の一言で完了 |
| 状態によって成否が変わる | 毎回状態チェックして分岐 |

### Skillsが向いているケース

- **手順が固定的で、LLMの判断を入れたくない作業**（ビルド、デプロイ、環境構築など）
- **複数ステップにまたがり、各ステップの成否確認が必要な作業**
- **繰り返し実行する定型作業**

逆に、コードレビューやリファクタリングのようなLLMの判断力が活きる作業には向きません。**AIに任せる部分と、固定化する部分を分ける**のが、Claude Codeを安定して使うコツだと感じています。
